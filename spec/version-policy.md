## 3. Pre-release 태그

### 3.1 태그 종류

| 태그 | 용도 | 안정성 | 예시 |
|------|------|--------|------|
| `-alpha.N` | 내부 개발 버전 | 불안정 | `0.7.0-alpha.1` |
| `-beta.N` | 베타 테스트 버전 | 준안정 | `0.8.0-beta.3` |
| `-rc.N` | 릴리스 후보 | 안정 | `1.0.0-rc.1` |

### 3.2 태그 사용 규칙

```bash
# Alpha: 개발 중 내부 테스트
0.7.0-alpha.1 → 0.7.0-alpha.2 → ... → 0.7.0

# Beta: 베타 테스터 피드백 반영
0.8.0-beta.1 → 0.8.0-beta.2 → ... → 0.8.0

# RC: 최종 검증
1.0.0-rc.1 → 1.0.0-rc.2 → ... → 1.0.0
```

## 4. 단계별 출시 조건

### 4.1 MVP (0.7.x) 출시 조건

- [ ] 핵심 기능 구현 완료
- [ ] 기본 UI/UX 완성
- [ ] 내부 QA 통과
- [ ] 에러 핸들링 기본 구현
- [ ] 데모 가능 상태
- [ ] CLAUDE.md 작성 완료
- [ ] 기본 README.md 작성

### 4.2 Closed Beta (0.8.x) 조건

- [ ] MVP 피드백 반영 (80% 이상)
- [ ] 베타 테스터 온보딩 프로세스 구축
- [ ] 피드백 수집 시스템 구축 (Sentry, Hotjar 등)
- [ ] 기본 모니터링/로깅 (Vercel Analytics)
- [ ] 데이터 백업 체계 수립
- [ ] 베타 이용약관 준비

### 4.3 Open Beta (0.9.x) 조건

- [ ] Closed Beta 피드백 80% 반영
- [ ] 성능 최적화 완료 (Lighthouse 80+)
- [ ] 보안 점검 통과
- [ ] 사용자 가이드 문서화
- [ ] 고객 지원 체계 구축 (이메일, 채팅)
- [ ] 에러 트래킹 고도화

### 4.4 정식 출시 (1.0.0) 조건

- [ ] Open Beta 버그 리포트 100% 해결
- [ ] Lighthouse 점수 90+ (모든 지표)
- [ ] 테스트 커버리지 80%+
- [ ] SLA 정책 수립 및 공개
- [ ] 결제 시스템 연동 완료 (ideaonaction.ai)
- [ ] 마케팅 자료 준비 (랜딩페이지, 홍보물)
- [ ] 법적 문서 준비 (이용약관, 개인정보처리방침)

## 6. 버전 관리 프로세스

### 6.1 버전 업 워크플로우

```
1. 기능/수정 완료
        │
        ▼
2. 변경 유형 판단
   ├─ Breaking Change? → Major 업
   ├─ 새 기능?        → Minor 업
   └─ 버그 수정?      → Patch 업
        │
        ▼
3. CHANGELOG.md 업데이트
        │
        ▼
4. package.json 버전 업데이트
        │
        ▼
5. Git 태그 생성
   git tag -a v0.7.1 -m "버전 설명"
        │
        ▼
6. GitHub Release 생성
```

### 6.2 버전 태그 규칙

```bash
# 버전 태그 생성
git tag -a v0.7.0 -m "MVP: 초기 릴리스"
git push origin v0.7.0

# Pre-release 태그
git tag -a v0.8.0-beta.1 -m "Closed Beta: 첫 번째 빌드"
git push origin v0.8.0-beta.1
```
